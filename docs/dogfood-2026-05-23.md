## Two real drifts surfaced and fixed

### Drift A — UI's EIP-712 had 8 fields, contract has 9 (`maxFee` missing)

The Trade UI's `usePlaceOrder` hook signed against
`packages/perps/src/typed-data.ts::SIGNED_ORDER_TYPES`, which had only 8
fields. The on-chain `FxOrderSettlement.SIGNED_ORDER_TYPEHASH` has 9
(extra `maxFee: uint256` at position 5). The Rust matcher's
`matcher-types/src/eip712.rs` also has 9.

Net: every intent the UI signed produced an 8-field EIP-712 digest;
neither the matcher (`intent_translator → SignerMismatch`) nor the
contract (`_validateOrder → InvalidSignature`) would recover the
correct signer. The Trade UI's submit path was **structurally broken
end-to-end** since day one — never noticed because the TS
`apps/keeper-perps-matcher` used the same broken schema and nothing
ever actually settled from the UI.

**Fix:** added `maxFee: uint256` as the 5th field in
`SIGNED_ORDER_TYPES` + matching `maxFee: bigint` in
`SignedOrderMessage`, hard-coded to 0n in `buildSignedOrderMessage`.
Mirrors what the Rust matcher's `intent_translator` already does
(`maxFee: U256::ZERO` per the spec comment). Commit `0b65fcf`.

### Drift B — `packages/contracts` Arc perps addresses were pre-sprint-1

Every address in `packages/contracts/src/index.ts::CONTRACTS[5042002].perps`
(and `telarana.fxOracle`) was stale — predated the sprint-1 broadcast
on 2026-05-21. Examples:

| FxFundingEngine | `0x725822e8BC6edbcBa52914149e25f2671290C6D2` | `0x859bA11A3693895f8B03C31C6AE3b8F04992115B` |

| FxLiquidationEngine | `0x01f71c1E74350633bBC9d554ca35DA40412DCFB7` | `0xF579e265EF1D5E67EfDbb1F20863465E94a9d3eA` |

The API uses `loadContracts()` → these stale addresses for the
EIP-712 `verifyingContract` field. The matcher reads
`perp-stack-5042002.json` for its own verifier. They were targeting
different contracts; any intent signed against the stale address would
be rejected by the matcher (different EIP-712 domain → different
digest → `SignerMismatch`).

**Fix:** updated `packages/contracts/src/index.ts::CONTRACTS[5042002].perps`
+ `.telarana.fxOracle` to the live sprint-1 addresses, citing
`~/coding-dojo/fx-telarana/deployments/perp-stack-5042002.json` as the
source of truth. The `bun run contracts:sync` script only syncs ABIs,
not addresses — those are hand-maintained per chain.

**Follow-up TODO:** the `contracts:sync` script should also pull
addresses from the deployment manifests so this drift doesn't recur.
File as Phase 8.

## What we couldn't demonstrate in headless

- **SSE status pill animation in the Trade UI** — the
  `useIntentStatusStream` hook only fires when `lastIntentId` is set,
  which happens after `usePlaceOrder.mutateAsync` resolves. That
  requires a connected wallet (Dynamic.xyz modal). The headless
  browser can't easily complete Dynamic auth (OAuth / email). Verified
  separately via curl against `GET /perps/intents/:id/stream` —
  `event: connected → status → terminal` works end-to-end. UI hook
  subscribes correctly (typecheck + code review).
- **Actual settlement** — the EURC/USDC market's on-chain
  `maxOpenInterest` is currently 0.001 USDC. Any non-trivial intent
  trips `OiCapBreach`. The LP gate is doing exactly what it should;
  this is an operator-config concern upstream (fx-telarana sets the
  cap via `FxPerpClearinghouse.configureMarket`). To actually fill, the
  operator needs to either raise the cap or test a smaller intent
  (under the existing skew).
